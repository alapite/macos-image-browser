<task type="auto">
  <name>Task 1: Add scan instrumentation (duration + counts) in AppState</name>
  <files>AppState.swift, Logging.swift</files>
  <action>
Introduce a small logging utility using Apple frameworks only:
- Create `Logging.swift` that exposes one or more `os.Logger` instances (e.g., `Logger.scan`, `Logger.imageLoad`) with a consistent subsystem/category.

Instrument folder scanning:
- Surround the folder scan (the work currently in `loadImages(from:)`) with duration measurement.
- Log key metrics at info level:
  - selected folder path (or lastPathComponent to reduce noise)
  - total images discovered
  - number of failures (`failedImages.count`)
  - elapsed time

Optional (nice if low effort):
- Use `OSSignposter` to create signposts around scan so Instruments can visualize it.

Avoid noisy per-file logs; keep logs coarse-grained so scanning large folders doesn't spam output.
  </action>
  <verify>`swift build` succeeds</verify>
  <done>
Scan produces a single structured log entry per scan with count + duration, and build remains dependency-free.
  </done>
</task>

<task type="auto">
  <name>Task 2: Add image load instrumentation (decode time + failures) in the SwiftUI layer</name>
  <files>ContentView.swift, Logging.swift</files>
  <action>
Add a small seam in `ContentView.swift` so image decoding is not done inline with no observability:
- Wrap `NSImage(contentsOf:)` calls (main image + thumbnails) in a helper that:
  - measures decode time
  - logs failures (nil image) with URL
  - keeps logs sampled/coarse to avoid per-frame spam

Constraints:
- Do not introduce a full caching or async loader here (that is Phase 3); only add observability around the current behavior.
- Keep SwiftUI code readable: prefer extracted helpers or small view-model helpers rather than long modifier chains with embedded logging.
  </action>
  <verify>
`swift build` passes; `swift run ImageBrowser` and load a folder, then confirm logs appear in Console.app (filter by subsystem/category).
  </verify>
  <done>
Main image and thumbnail loads have at least coarse timing/failure logs, enabling later before/after comparisons when caching/async decode is added.
  </done>
</task>
